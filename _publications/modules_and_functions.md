## Մոդուլներ և ֆունկցիաներ

Ֆունկցիաներ մենք նախկինում էլ ենք օգտագործել. `IO.puts`, `Map.get`, `Enum.sum` ... Իսկ մեր `.exs` ընդլայնումով ֆայլերի օրինակներում ամեն անգամ հանդիպում էին `defmodule`-ն ու `def`-ը՝ խոստանալով, որ «կբացատրենք ավելի ուշ»։ Այդ «ավելի ուշ»-ը եկել է։ Ֆունկցիաները Elixir-ում այնքան մեծ թեմա են, որ նվիրելու ենք մի ամբողջ հոդվածաշար, և սա դրա առաջին մասն է՝ մոդուլներն ու `named function`-ները(Հայերեն երևի թե կարելի է թարգմանել անվանակիր ֆունկցիաներ)։

### Ինչո՞ւ են պետք մոդուլները

Elixir-ում ամեն ֆունկցիա «տուն» ունի. ֆունկցիաները ապրում են մոդուլների մեջ։ Մոդուլը իրար հետ տրամաբանորեն կապված ֆունկցիաների խումբ է՝ մեկ անվան տակ։ Մենք դա արդեն տեսել ենք․ `Map.get`-ի մեջ `Map`-ը մոդուլն է, `get`-ը՝ նրա ֆունկցիան։ `Enum`, `Tuple`, `String`՝ այս բոլորը մոդուլներ են, որոնց ֆունկցիաներն արդեն օգտագործել ենք։

Եթե գալիս եք JavaScript-ից, մոդուլը կհիշեցնի ֆայլ-մոդուլը՝ իր export-արած ֆունկցիաներով։ Java-ի կամ C#-ի աշխարհից եկածների համար ամենամոտ պատկերը static մեթոդներով class-ն է. object չենք ստեղծում, ուղղակի կանչում ենք `Մոդուլ.ֆունկցիա(...)`։

Անվանման կանոնները խիստ են. մոդուլների անունները գրվում են մեծատառով՝ `PascalCase` ոճով (`MyModule`), ֆունկցիաներինը՝ փոքրատառ `snake_case`-ով (`my_function`)։ Ի դեպ մոդուլի անունը իրականում սովորական [atom](./atoms.md) է՝ `is_atom(Map)`-ը վերադարձնում է `true`։

### Ինչպես ստեղծել մոդուլ և հայտարարել ֆունկցիա

Մոդուլը ստեղծվում է `defmodule`-ով, ֆունկցիան՝ `def`-ով, և երկուսն էլ իրենց մարմինը ընդգրկում են `do ... end` բլոկի մեջ.

```elixir
iex> defmodule Greeter do
...>   def hello(name) do
...>     "Hello, " <> name
...>   end
...> end
{:module, Greeter, <<70, 79, 82, 49, ...>>, {:hello, 1}}
iex> Greeter.hello("Ani")
"Hello, Ani"
```

`<>` օպերատորը string-երը կպցնում է իրար(concatenation)։ Pattern matching-ի հոդվածում այն string էր բաժանում, այստեղ՝ միացնում է. նույն օպերատորը, երկու կիրառություն, ինչպես list-երի դեպքում `|`-ը։

Եթե iex-ում նույն մոդուլը երկրորդ անգամ սահմանեք, Elixir-ը կզգուշացնի՝ «redefining module Greeter»։ Դա սխալ չէ, ուղղակի հիշեցում է, որ հին տարբերակը փոխարինվեց նորով։

### Ֆունկցիայի վերադարձրած արժեքի կանոնը

Ուշադիր նայեք `hello` ֆունկցիային։ JavaScript-ում կգրեինք.

```javascript
function hello(name) {
  return "Hello, " + name;
}
```

Elixir-ում `return` բանալի բառ **գոյություն չունի**։ Կանոնը մեկն է և բացառություն չունի. **ֆունկցիան վերադարձնում է իր մարմնի վերջին արտահայտության արժեքը**։ Միշտ։ Ամեն ֆունկցիա ինչ-որ բան վերադարձնում է, նույնիսկ եթե դուք դրա մասին չեք մտածել.

```elixir
iex> defmodule Circle do
...>   def area(radius) do
...>     pi = 3.14159
...>     pi * radius * radius
...>   end
...> end
iex> Circle.area(10)
314.159
```

Առաջին տողը միջանկյալ հաշվարկ է, երկրորդը՝ վերջինն է, ուրեմն հենց դա է պատասխանը։ Սրանից բխում է նաև, որ «կես ճանապարհից վերադառնալ» (early return) Elixir-ում չկա. ֆունկցիան միշտ հասնում է իր վերջին տողին։ Սա կարող է մի փոքր անսովոր թվալ, սակայն հարմարվելուց հետո կտեսնենք նաև առավելությունները․ կոդն ավելի կանխատեսելի է. պատասխանը միշտ ֆունկցիայի ներքևում է։

### Սեղմ գրելաձև

Կարճ ֆունկցիաների համար `do ... end`-ը երկար է։ Կա սեղմ գրելաձև՝ `, do:`.

```elixir
iex> defmodule Circle2 do
...>   def area(radius), do: 3.14159 * radius * radius
...> end
iex> Circle2.area(1)
3.14159
```

Երկու ձևերը բացարձակ համարժեք են։ Պաշտոնական փաստաթղթերում և կոդում սեղմ գրելաձևը հաճախ է հանդիպում, այնպես որ ցանկալի է աչքը վարժեցնել այն տեսնելու և ընկալելու։ 

### Arity. ֆունկցիայի «ազգանունը»

Պաշտոնական փաստաթղթերում ֆունկցիաների անուններին կպած մի թիվ ենք միշտ տեսնում. `length/1`, `Map.get/3`, `elem/2`։ Այդ թիվը կոչվում է **arity**՝ ֆունկցիայի ընդունած արգումենտների քանակը։

Ինչո՞ւ է դա այդքան կարևոր, որ գրվում է անվան կողքին։ Որովհետև Elixir-ում **նույն անունով, բայց տարբեր arity-ով ֆունկցիաները բոլորովին տարբեր ֆունկցիաներ են**.

```elixir
iex> defmodule Greeter2 do
...>   def hello do
...>     "Hello, world"
...>   end
...>
...>   def hello(name) do
...>     "Hello, " <> name
...>   end
...> end
iex> Greeter2.hello()
"Hello, world"
iex> Greeter2.hello("Ani")
"Hello, Ani"
```

Սա ուղղակի overloading-ի պես մի բան չէ, որ «նույն ֆունկցիան է՝ տարբեր պարամետրերով»։ `hello/0`-ն և `hello/1`-ը երկու տարբեր ֆունկցիա են, բայց ունեն նույն անունը։ Դրա համար էլ Elixir-ի աշխարհում ֆունկցիային միշտ դիմում են լրիվ «անուն-ազգանունով»՝ `hello/1`։

Վիրտուալ մեքենան ամեն ֆունկցիա նույնականացնում է երեք բաղադրիչով՝ մոդուլ, անուն, arity (այսպես կոչված MFA՝ module-function-arity եռյակը)։ `Greeter2.hello/0`-ն և `Greeter2.hello/1`-ը մոդուլի ֆունկցիաների աղյուսակում երկու առանձին գրառում են։ Java-ի ու C#-ի overload resolution-ի նման ընտրության քայլ այստեղ գոյություն չունի. ընտրելու բան չկա, երբ arity-ն անվան մասն է։

Իսկ երբ arity-ն սխալ է, սխալի հաղորդագրությունը ոչ միայն ասում է «չկա», այլև հուշում է, թե ինչ կա

```elixir
iex> Greeter2.hello("Ani", "Aram")
** (UndefinedFunctionError) function Greeter2.hello/2 is undefined or private. Did you mean:

      * hello/0
      * hello/1
```

### Default արժեքներ

Ֆունկցիաների պարամետրերին default արժեք տալու համար օգտագործվում է `\\` նշանը.

```elixir
iex> defmodule Greeter3 do
...>   def hello(name \\ "world") do
...>     "Hello, " <> name
...>   end
...> end
iex> Greeter3.hello()
"Hello, world"
iex> Greeter3.hello("Ani")
"Hello, Ani"
```

Եվ մի դետալ. այս մեկ `def`-ից Elixir-ն իրականում ստեղծում է **երկու** ֆունկցիա՝ `hello/0` և `hello/1`։

### def-ը public է, defp-ն՝ private

Java-ում և C#-ում մեթոդի տեսանելիությունը որոշում է բանալի բառը՝ `public` կամ `private`։ Elixir-ում նույն ընտրությունն արվում է սահմանման ձևով. `def`-ը ստեղծում է public ֆունկցիա, որը կանչելի է մոդուլից դուրս, իսկ `defp`-ն (p-ն հենց private-ի սկզբնատառն է)՝ private ֆունկցիա, որը հասանելի է միայն նույն մոդուլի ներսում.

```elixir
iex> defmodule Store do
...>   def final_price(price) do
...>     price - discount(price)
...>   end
...>
...>   defp discount(price) do
...>     price * 0.1
...>   end
...> end
iex> Store.final_price(200)
180.0
iex> Store.discount(200)
** (UndefinedFunctionError) function Store.discount/1 is undefined or private
```

Մոդուլի ներսից `discount/1`-ը հանգիստ կանչվում է, դրսից՝ գոյություն չունի։ Ուշադրություն դարձրեք սխալի տեքստին. «undefined **or private»**։ Սխալի հաղորդագրության մեջ դիտավորյալ չի ասվում, թե կոնկրետ որն է. private ֆունկցիան դրսի համար նույնքան անտեսանելի է, որքան գոյություն չունեցողը։

private ֆունկցիաների գոյությունն ապահովում է encapsulation-ը, որին մենք ծանոթ ենք OOP հայեցակարգ ունեցող ծրագրավորման լեզուներից՝ օրինակ Java-ից կամ C#-ից։ Մոդուլի public ֆունկցիաները նրա ինտերֆեյսն են՝ պայմանագիր բոլոր նրանց հետ, ովքեր օգտագործում են մոդուլը։ Private ֆունկցիաներն իրականացման մանրամասներ են. դրանք վաղը կարող ենք վերանվանել, վերաշարադրել կամ ջնջել՝ վստահ, որ մոդուլից դուրս ոչ մի կոդ չի ջարդվի։ Փոքր մոդուլում այս սահմանը կարող է մանրուք թվալ, բայց աճող մոդուլում այն չափազանց մեծ դեր ունի և հնարավորություն է տալիս գրել մաքուր, հասկանալի, հեշտ փոփոխելի կոդ։

### .ex, .exs և կոմպիլյացիան

Մեր բոլոր օրինակների ֆայլերը `.exs` վերջավորությամբ էին և աշխատում էին `elixir file_name.exs` հրամանով։ Ինչո՞ւ

- `.exs`՝ **script**֊ային ֆայլեր են. Elixir-ը կարդում է և տեղում կատարում։ Իդեալական է սովորելու, փորձերի և փոքր խնդիրների համար՝ հենց այն, ինչ անում էինք։
- `.ex`՝ իսկական նախագծերի ֆայլերն են. դրանք **կոմպիլացվում** են՝ վերածվում `.beam` ֆայլերի, որոնք արդեն պատրաստ են գործարկվելու `BEAM` վիրտուալ մեքենայի վրա։ 

Իրական նախագծերում `.ex` ֆայլերը, կոմպիլյացիան ու շատ այլ բաներ կառավարում է `mix` գործիքը, որին կծանոթանանք առաջիկայում։

### Փորձեք ինքներդ

Բացեք iex-ը և սահմանեք `Calculator` մոդուլ՝ `add/2` ֆունկցիայով, որը գումարում է երկու թիվ։ Կանչեք այն երեք արգումենտով և ուշադիր կարդացեք սխալի հաղորդագրությունը՝ ի՞նչ է հուշում Elixir-ը։ Ապա `add`-ին տվեք default արժեք երկրորդ արգումենտի համար (`b \\ 10`) և ստուգեք, որ հիմա և՛ `add(5)`-ն է աշխատում, և՛ `add(5, 20)`-ը։ Դրանից հետո ավելացրեք `defp double(x)` ֆունկցիա և public `quadruple/1`, որը երկու անգամ կանչում է `double`-ը. համոզվեք, որ `quadruple`-ը դրսից աշխատում է, իսկ `double`-ը՝ ոչ։

### Ի՞նչ է սպասվում առջևում

Այս հոդվածում ֆունկցիաներից ամեն մեկն «ապրում էր» իր մոդուլում և ուներ անուն։ Հաջորդ մասում կհանդիպենք բոլորովին այլ տեսակի ֆունկցիաների՝ **անանուն ֆունկցիաներին**, որոնք կարող են գոյություն ունենալ մոդուլից դուրս, պահվել փոփոխականների մեջ և փոխանցվել այլ ֆունկցիաների որպես սովորական արժեք։
