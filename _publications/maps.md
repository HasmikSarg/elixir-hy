## Map տվյալների տիպը

[Նախորդ հոդվածում](./lists.md) ծանոթացանք list-երին, որտեղ տարրը գտնում ենք իր դիրքով՝ առաջինը, երկրորդը, երրորդը և այլն։ Բայց շատ հաճախ արժեքը պետք է գտնել ոչ թե դիրքով, այլ անունով։ Պատկերացրեք հեռախոսագիրք. ձեզ պետք է կոնկրետ մարդու համարը, և բոլորովին կարևոր չէ, թե նա ցուցակի քանիերորդ տողում է։ Հենց դրա համար էլ գոյություն ունի map-ը, որը հանդիսանում է Elixir-ի հիմնական «բանալի-արժեք» (key-value) կառուցվածքը։

Անվանումը թող չշփոթեցնի. map-ը քարտեզի հետ կապ չունի։ Ավելի դիպուկ պատկերը բառարանն է. բառով (բանալիով) գտնում ենք բացատրությունը (արժեքը)։ Եթե գալիս եք Java-ից կամ C#-ից՝ map-ը ձեզ կհիշեցնի HashMap-ն ու Dictionary-ն։

### Ինչպե՞ս ստեղծել map

Map-ը ստեղծվում է `%{}` գրելաձևով, իսկ բանալի-արժեք զույգերը գրվում են `բանալի => արժեք` տեսքով.

```elixir
iex> %{}
%{}
iex> %{"one" => :two, 3 => "four"}
%{3 => "four", "one" => :two}
```

Ուշադրություն դարձրեք երկրորդ օրինակին. մենք `"one"`-ը գրեցինք առաջինը, բայց iex-ը այն տպեց երկրորդը։ Սա սխալ չէ. map-ի զույգերը **հերթականություն չունեն**։ List-ը հաջորդականություն էր, map-ը՝ ոչ։ Երբեք մի՛ նախագծեք ծրագրի տրամաբանությունը այն հույսով, թե map-ի զույգերը կմնան ձեր գրած հերթականությամբ։

Բանալի կարող է լինել ցանկացած արժեք՝ string, atom, թիվ, նույնիսկ մեկ այլ list կամ map։ Արժեքը՝ նույնպես ցանկացած տիպի։

Մի map-ում նույն բանալին երկու անգամ լինել չի կարող։ Եթե գրելիս կրկնել եք բանալին, կմնա վերջինը, իսկ Elixir-ը կզգուշացնի, որ հավանաբար սխալվել եք.

```elixir
iex> %{"a" => 1, "a" => 2}
%{"a" => 2}
```

### Atom բանալիների կրճատ գրելաձևը

Գործնականում map-երի բանալիները հաճախ [atom-ներ](./atoms.md) են։ Այդ դեպքի համար կա համառոտ ու հարմար գրելաձև.

```elixir
iex> %{a: 1, b: 2}
%{a: 1, b: 2}
```

Սա ճիշտ նույնն է, ինչ `%{:a => 1, :b => 2}`։ Երբ բոլոր բանալիները atom են, Iex-ը պատասխանը միշտ ցույց է տալիս կրճատ ձևով։

Երկու գրելաձևերը կարելի է խառնել, բայց մի պայմանով՝ կրճատ ձևը պետք է լինի վերջում.

```elixir
iex> %{"hello" => "world", a: 1, b: 2}
%{:a => 1, :b => 2, "hello" => "world"}
```

Նկատեցի՞ք տարօրինակությունը. քանի որ այստեղ ոչ բոլոր բանալիներն են atom, iex-ը ամեն ինչ տպեց լրիվ `=>` ձևով։

### Ինչպե՞ս կարդալ արժեքը

Ամենապարզ ձևը քառակուսի փակագծերն են.

```elixir
iex> capitals = %{"Armenia" => "Yerevan", "France" => "Paris"}
%{"Armenia" => "Yerevan", "France" => "Paris"}
iex> capitals["Armenia"]
"Yerevan"
iex> capitals["Spain"]
nil
```

Չեղած բանալու դեպքում ստանում ենք `nil`՝ արժեքի բացակայություն նշանակող հատուկ արժեքը։ Ծրագիրը չի ընդհատվում, պարզապես ստանում ենք «ոչինչ»։

`Map.get/3` ֆունկցիան նույնն է անում, բայց թույլ է տալիս երրորդ արգումենտով նշել, թե ինչ վերադարձնել, երբ բանալին չկա.

```elixir
iex> Map.get(capitals, "Spain")
nil
iex> Map.get(capitals, "Spain", "unknown")
"unknown"
```

Իսկ երբ բանալիները atom են, կա նաև երրորդ՝ կետով գրելաձևը.

```elixir
iex> user = %{name: "Ani", age: 25}
%{age: 25, name: "Ani"}
iex> user.name
"Ani"
iex> user.email
** (KeyError) key :email not found in: %{age: 25, name: "Ani"}
```

Եվ ահա առաջին լուրջ տարբերությունը. `user[:email]`-ը կվերադարձներ `nil`, իսկ `user.email`-ը ծրագիրն ընդհատեց `KeyError` սխալով։

Ո՞րն ընտրել։ Կանոնը պարզ է.

- `map[key]`՝ երբ բանալին կարող է չլինել, և դա նորմալ է (օրինակ՝ օգտատերը կարող է հասցե նշած լինել կամ չլինել)։
- `map.key`՝ երբ բանալին պարտադիր պետք է լինի։ Այս դեպքում սխալը մեզ օգնում է. ավելի լավ է այն երևա հենց այդ տողում, քան `nil`-ը լուռ շարունակի ճանապարհը և ծրագիրն ընդհատվի բոլորովին այլ տեղում՝ դժվար գտնվող սխալով։

Երկրորդ մոտեցման վրա են կառուցված struct-երը՝ «խիստ» map-երը, որոնց կծանոթանանք առաջիկա հոդվածներից մեկում։

### String, թե՞ atom բանալի

Այստեղ մի փոքրիկ նրբություն կա։ Հետևյալ երկու map-երը իրար հետ ոչ մի կապ չունեն.

```elixir
iex> %{"name" => "Ani"}
%{"name" => "Ani"}
iex> %{name: "Ani"}
%{name: "Ani"}
```

Առաջինի բանալին `"name"` string-ն է, երկրորդինը՝ `:name` atom-ը։ Երկու տարբեր արժեք, ուրեմն՝ երկու տարբեր բանալի.

```elixir
iex> user = %{name: "Ani"}
%{name: "Ani"}
iex> user[:name]
"Ani"
iex> user["name"]
nil
```

Գործնական խորհուրդ. դրսից եկած տվյալները (օրինակ՝ JSON) գրեթե միշտ string բանալիներով են, ուրեմն դրանց դիմեք `data["name"]` ձևով։ Ձեր սեփական կոդում ստեղծած map-երի համար atom բանալիներն ամենահարմարն են։

### Բանալիների խիստ համեմատությունը

Եվս մեկ նրբություն. map-ը բանալիները համեմատում է խստորեն՝ առանց տիպերի փոխակերպման։ [Թվային տիպերի հոդվածից](./numeric_types.md) հիշում ենք, որ 1-ը ամբողջ թիվ է, իսկ 1.0-ը՝ float։ Թվաբանության մեջ դրանք հավասար են, բայց map-ի համար՝ երկու բոլորովին տարբեր բանալի.

```elixir
iex> m = %{1 => "one"}
%{1 => "one"}
iex> m[1]
"one"
iex> m[1.0]
nil
```

### Ինչպե՞ս թարմացնել արժեքը

Գոյություն ունեցող բանալու արժեքը փոխելու համար կա հատուկ գրելաձև՝ `|` նշանով.

```elixir
iex> user = %{name: "Ani", age: 25}
%{age: 25, name: "Ani"}
iex> %{user | age: 26}
%{age: 26, name: "Ani"}
iex> user
%{age: 25, name: "Ani"}
```

Երկու կարևոր բան կատարվեց այստեղ։

Առաջին. նայեք վերջին տողին՝ `user`-ը մնաց անփոփոխ։ Elixir-ում տվյալները երբեք չեն փոխվում. `%{user | age: 26}`-ը ստեղծեց **նոր** map, ճիշտ ինչպես `Enum.reverse`-ը list-երի հոդվածում նոր list էր վերադարձնում՝ անփոփոխ թողնելով հինը։ Եթե հին map-ը այլևս պետք չէ, թարմացրածը «վերագրեք» նույն փոփոխականին։. `user = %{user | age: 26}`։ «Վերագրել»֊ը նշել եմ չակերտների մեջ, զուտ մեյնսթրիմային լեզուներից եկած ծրագրավորողներին հասկանալի լինելու համար։ Իրականում Elixir-ում `=` օպերատորը վերագրում չի կատարում, այն արժեքը կապում է փոփոխականի հետ (binding), որը կոնցեպտուալ առումով տարբերվում է մեր իմացած վերագրումից։

Երկրորդ. `|` գրելաձևը միայն **թարմացնում** է, ավելացնել չի կարող.

```elixir
iex> %{user | email: "ani@example.com"}
** (KeyError) key :email not found in: %{age: 25, name: "Ani"}
```

Առաջին հայացքից սա անհարմարություն է թվում, բայց իրականում անհրաժեշտ պաշտպանություն է։ Պատկերացրեք՝ ուզում էիք թարմացնել `age`-ը, բայց վրիպակով գրեցիք `agee`։ Շատ լեզուներ լուռ կավելացնեին նոր `agee` բանալի, և սխալը երկար կմնար աննկատ։ Elixir-ը վրիպակը բռնում է տեղում։

Իսկ երբ իսկապես նոր զույգ է պետք ավելացնել, կա `Map.put/3` ֆունկցիան.

```elixir
iex> Map.put(user, :email, "ani@example.com")
%{age: 25, email: "ani@example.com", name: "Ani"}
```

`Map.put`-ը համապիտանի է. եթե բանալին չկա՝ ավելացնում է, եթե կա՝ թարմացնում է արժեքը։ Նշենք նաև, որ `|`-ը միայն atom-ների դեպքում չէ, որ աշխատում է. `=>` ձևով նույն կերպ կարող եք թարմացնել ցանկացած տիպի բանալու արժեք։

### Շարունակենք ծանոթությունը pattern matching-ի հետ

Հիշո՞ւմ եք [bitstring-երի հոդվածի](./bitstrings_and_binaries.md) վերջին վարժությունը, որտեղ `<<r, g, b>>` գրելաձևով գույնը «բաժանեցինք» երեք մասի։ Այդ կախարդանքի անունը pattern matching է։ Ամբողջական բացատրությունը դեռ առջևում է, բայց map-երի հետ աշխատող ամենաօգտակար կտորը կարելի է սովորել հենց հիմա։

Գաղտնիքը `=` օպերատորի մեջ է։ Elixir-ում այն ոչ թե «վերագրում» է, այլ **համապատասխանեցում**. ձախ կողմում կաղապարն է, աջ կողմում՝ արժեքը, և Elixir-ը փորձում է դրանք իրար համապատասխանեցնել։ Երբ ձախում պարզ անուն է (`x = 1`), համապատասխանեցումը միշտ հաջողվում է, և թվում է, թե սովորական վերագրում է։ Բայց ձախում կարող է լինել նաև map-ի կաղապար.

```elixir
iex> %{name: n} = %{name: "Ani", age: 25}
%{age: 25, name: "Ani"}
iex> n
"Ani"
```

Կաղապարում նշում ենք միայն մեզ հետաքրքրող բանալիները, և Elixir-ը դրանց արժեքները «կապում» է մեր տված անունների հետ։ Մնացած բանալիները (այստեղ՝ `age`) անտեսվում են։

Այստեղից մի անսպասելի հետևանք. դատարկ `%{}` կաղապարը ոչ մի բանալի չի պահանջում, ուրեմն համընկնում է **ցանկացած** map-ի հետ.

```elixir
iex> %{} = %{name: "Ani"}
%{name: "Ani"}
```

Այսինքն՝ `%{} = ...` գրելաձևը երբեք չի ստուգում, թե map-ը դատարկ է. այն ստուգում է միայն, որ արժեքը map է։

Իսկ երբ կաղապարի բանալին map-ում չկա, ստանում ենք `MatchError`.

```elixir
iex> %{email: e} = %{name: "Ani"}
** (MatchError) no match of right hand side value: %{name: "Ani"}
```

Առայժմ այսքանը բավական է։ Pattern matching-ը Elixir-ի ամենահզոր գործիքներից է և արժանի է առանձին, հանգամանալից հոդվածի. երբ ծանոթ լինենք ևս մի քանի տվյալների տիպի, մեկ հոդվածով կամբողջացնենք այն՝ արդեն ծանոթ բոլոր տիպերի օրինակներով։

### Չափը և արագությունը

Զույգերի քանակը տալիս է `map_size/1` ֆունկցիան.

```elixir
iex> map_size(%{a: 1, b: 2})
2
```

Հետաքրքիր մանրուք. list-երի հոդվածում տեսանք, որ `length/1`-ը O(n) է, քանի որ ստիպված է հաշվել տարրերը մեկ առ մեկ։ `map_size/1`-ը O(1) է.

Օգտակար են նաև `Map.keys/1` և `Map.values/1` ֆունկցիաները, որոնք վերադարձնում են բանալիների ու արժեքների list-երը.

```elixir
iex> Map.keys(%{a: 1, b: 2})
[:a, :b]
iex> Map.values(%{a: 1, b: 2})
[1, 2]
```

Սրանք, բնականաբար, O(n) են. ամբողջ list կառուցելու համար պետք է անցնել բոլոր զույգերով։

Իսկ ամենակարևոր հարցը՝ որքա՞ն արագ է map-ը գտնում բանալին։ List-ում արժեք փնտրելը O(n) է. ինչքան երկար է list-ը, այնքան երկար է որոնումը։ Map-ը այլ կերպ է կառուցված. նույնիսկ երբ զույգերի քանակը հազարներով է աճում, բանալիով որոնման ժամանակը աննշան է փոխվում։ Ճշգրիտ լինելու համար՝ դա O(1) չէ, բայց O(n)-ից անհամեմատ լավ է. թե կոնկրետ ինչ է, կպարզենք Big O-ին նվիրված հոդվածում։ Առայժմ հիշեք գործնական կանոնը. երբ արժեքները բանալիով եք փնտրելու, map-ն է ճիշտ ընտրությունը։

Ի դեպ, `Enum` մոդուլի ֆունկցիաները, որոնց հանդիպել ենք list-երի օրինակներում (`Enum.sum`, `Enum.reverse`), աշխատում են նաև map-երի հետ. դրանց մանրամասն կանդրադառնանք առաջիկայում։

### Փորձեք ինքներդ

Սկսեք iex-ից. ստեղծեք ձեր ընկերների տարիքների map-ը (անունը՝ string բանալի, տարիքը՝ արժեք)։ Կարդացեք մեկի տարիքը `[]`-ով, ապա փորձեք չեղած անուն և ստացեք `nil`։ Նույնը կրկնեք `Map.get/3`-ով՝ ձեր ընտրած default արժեքով։ Թարմացրեք մեկի տարիքը `|` գրելաձևով, նոր ընկեր ավելացրեք `Map.put`-ով, և ամեն քայլից հետո ստուգեք և՛ `map_size`-ը, և՛ սկզբնական map-ը՝ համոզվելու, որ այն մնացել է անփոփոխ։

### Ի՞նչ է սպասվում առջևում

Առաջիկա հոդվածներում կշարունակենք ծանոթանալ Elixir-ի տվյալների տիպերին, իսկ երբ դրանք բավական լինեն, առանձին հոդվածով կամբողջացնենք pattern matching-ը՝ արդեն ծանոթ բոլոր տիպերի օրինակներով։
